## 1. System prompt vs user message

Когда ты вызываешь Claude API, ты отправляешь два типа текста: system
prompt и messages (user/assistant).

**User message** — это конкретная задача. «Извлеки данные из этого
текста», «Напиши функцию», «Ответь на вопрос». Это то, что в обычном
чате пользователь печатает в поле ввода.

**System prompt** — это инструкция модели «кем быть». Она задаёт роль,
тон, ограничения, формат ответа. System prompt не виден «пользователю»
в чате — это закулисная настройка.

На Day 02 ты это увидел буквально: один и тот же вопрос + три разных
system prompt-а («нейтральный», «тёрсий сеньор-инженер», «скептичный
YC-партнёр») дали три ответа, которые отличались не только длиной и
тоном, но и **содержанием**. «Тёрсий инженер» не просто ответил
короче — он посчитал другие вещи важными. Он выбросил context, который
«нейтральный» считал ключевым.

Почему так: LLM генерирует каждый следующий токен, опираясь на всё,
что видит в контексте. System prompt — это первое, что модель «читает».
Он задаёт фрейм, через который интерпретируется всё остальное. Если
фрейм «ты скептичный инвестор», модель буквально считает по-другому,
какая информация релевантна вопросу. Это не «фильтр на готовый
ответ» — это изменение процесса генерации с самого начала.

Практический вывод: когда что-то не так с ответом Claude, **первое,
что стоит менять — system prompt**, не user message. 80% качества
ответа определяется system prompt-ом.

В коде это выглядит так:

```typescript
const msg = await client.messages.create({
  model: "claude-sonnet-4-6",
  max_tokens: 256,
  system: "Ты извлекаешь данные из текста. Отвечай только JSON.",  // ← system
  messages: [
    { role: "user", content: "Вот текст: ..." }  // ← user message
  ],
});
```

`system` — строка, отдельный параметр. `messages` — массив объектов
с `role` и `content`. В messages могут быть `user` и `assistant`
сообщения (для многошаговых диалогов), но system prompt — всегда
отдельно.

## 2. «Один sample из распределения»

Это, пожалуй, самое важное понимание про LLM, которое отличает
«промпт-инженера» от «человека, который спрашивает ChatGPT».

Когда LLM генерирует ответ, она не «знает правильный ответ и
выдаёт его». Она генерирует текст **токен за токеном**, и на каждом
шаге выбирает следующий токен из распределения вероятностей.
Упрощённо: модель смотрит на всё, что уже написано (system +
user + свой ответ до этого момента), и говорит: «следующее слово
с вероятностью 40% — "embeddings", 25% — "APIs", 15% — "evals",
...» И выбирает одно.

Параметр `temperature` контролирует, насколько «случайный» этот
выбор. При temperature=0 модель всегда берёт самый вероятный
токен (детерминистично, но скучно). При temperature=1 — более
разнообразно, но менее предсказуемо.

Что это значит на практике: **один и тот же prompt может дать
разные ответы**. Ты спрашиваешь «что мне учить?» — Claude
отвечает «embeddings». Спрашиваешь снова — «evals». Оба ответа
валидны, оба содержат сигнал, но ни один не «правильный» в
абсолютном смысле. Каждый ответ — это один sample из
распределения возможных ответов.

Аналогия: представь, что ты спрашиваешь у 100 сеньор-инженеров
«что учить начинающему AI-инженеру?» Каждый даст свой ответ.
Ни один не врёт, но ни один не полон. Чтобы понять реальную
картину, нужно опросить несколько и посмотреть на распределение:
что повторяется чаще, что упоминается один раз. LLM — это как
один инженер, но каждый раз немного другой.

Отсюда растут evals: **один прогон ничего не доказывает**. Если
ты прогнал prompt один раз и получил хороший ответ — это не значит,
что prompt хороший. Может, тебе повезло. Нужен набор кейсов (10,
50, 100) и статистика: «в 9 из 10 случаев модель возвращает
валидный JSON» — вот это уже сигнал.

## 3. XML теги в промптах

Claude (и другие LLM) обучены на огромном количестве текста, включая
HTML, XML, Markdown. Когда модель видит XML-подобную разметку в
промпте, она «понимает» её как структурную инструкцию — вот здесь
начинается блок, вот здесь заканчивается.

Без тегов:
```
Return only valid JSON. No explanation.
```

Claude может ответить:
```
Here's the extracted data:
{"title": "...", "budget_usd": 3000}
```

Он «послушался» (вернул JSON), но добавил текст перед ним.
`JSON.parse()` сломается.

С тегами:
```
Return ONLY a JSON object wrapped in <json></json> tags.
```

Claude ответит:
```
<json>{"title": "...", "budget_usd": 3000}</json>
```

Теперь regex `<json>([\s\S]*?)<\/json>` надёжно вытащит JSON, даже
если Claude добавит текст снаружи тегов (что маловероятно, но
возможно).

Почему это работает: XML теги создают **визуальную и семантическую
границу**. Модель обучена на миллионах HTML/XML-документов, где
содержимое тегов — это данные, а всё снаружи — разметка или текст.
Когда ты говоришь «верни JSON в <json> тегах», модель применяет
этот паттерн: внутри тегов — данные, снаружи — ничего (потому что
ты сказал «only»).

Этот приём масштабируется. Ты можешь использовать XML теги для
любой структуры:

```
<thinking>тут модель рассуждает</thinking>
<answer>тут финальный ответ</answer>
```

Или:

```
<feedback>критика текста</feedback>
<revised>исправленная версия</revised>
```

Модель будет уважать эти границы и не смешивать содержимое разных
секций.

Важно: это не «магия». Это prompt engineering technique, основанная
на том, как модель обучена. На Claude это работает очень стабильно.
На других моделях (GPT, Llama) — может работать хуже, потому что
обучающие данные и fine-tuning отличаются.

## 4. Три уровня evals по Хамелю

Хамель Хусейн (инженер, ранее работал в GitHub, сейчас консультант
по ML) в своём 50-минутном видео описывает три уровня проверки
качества LLM-системы. Порядок — от дешёвого к дорогому, от
простого к сложному.

### Уровень 1: Code assertions

Это обычные unit-тесты для LLM. Ты пишешь код, который проверяет
ответ модели:

- `JSON.parse()` — парсится ли вообще?
- `schema.safeParse()` — соответствует ли схеме?
- `output.includes("keyword")` — есть ли нужное слово?
- `output.length < 500` — не слишком ли длинный?
- `budget_usd === 3000` — правильное ли число?

Плюсы: быстро (миллисекунды), дёшево (ноль API-вызовов), бинарно
(pass/fail, без «ну вроде нормально»), детерминистично (один и тот
же ответ всегда даёт одну оценку), масштабируется (1000 кейсов за
секунды).

Минусы: не умеет оценивать «мягкие» вещи — тон, релевантность,
полноту. «Ответ вежливый?» — code assertion не скажет.

Это то, что ты строил на Day 03-04. `BriefSchema.safeParse()` — это
level 1 eval в чистом виде.

### Уровень 2: LLM-as-judge

Когда code assertions не хватает (нужно оценить тон, релевантность,
качество текста), ты берёшь **другую** LLM-модель и просишь её
оценить ответ первой. Например:

```
Rate this customer service response on empathy, 1-5:
<response>...</response>
Output only the number.
```

Важно: прежде чем доверять judge, его нужно **калибровать**. Ты
берёшь 50 примеров, размечаешь руками (ставишь свои оценки), потом
прогоняешь judge на тех же 50 примерах, и считаешь процент согласия.
Если judge согласен с тобой в 90% случаев — им можно пользоваться.
Если в 60% — judge бесполезен, его промпт нужно переписывать.

Плюсы: может оценивать субъективные вещи, масштабируется (быстрее
и дешевле людей).

Минусы: не бесплатен (каждая оценка — API-вызов), нужна калибровка,
может быть предвзят (модели склонны ставить высокие оценки себе
подобным).

Лучшая практика: judge-модель ≠ генерирующая модель. Если ответ
генерирует claude-sonnet-4-6, judge должен быть другой (claude-opus-4-6 или
вообще GPT-4). Иначе ты проверяешь «нравится ли Claude ответ
Claude» — это не eval, это эхо-камера.

### Уровень 3: Human eval

Эксперт (или обычный пользователь) читает ответ модели и ставит
оценку. Это ground truth — самая точная оценка, потому что конечный
потребитель ответа — человек.

Плюсы: максимальная точность, ловит нюансы, которые не видит ни
код, ни LLM-judge.

Минусы: медленно (минуты на каждый пример), дорого (платишь людям),
не масштабируется для ежедневных прогонов.

Используется точечно: как калибровочный набор для уровней 1 и 2,
или для финальной приёмки перед релизом.

### Почему порядок важен

Большинство команд прыгают сразу на уровень 2 или 3, потому что
это «кажется правильным» — ведь мы же оцениваем текст, а не числа.
Но Хамель настаивает: **исчерпай уровень 1 перед уровнем 2**.

Причина: удивительно много ошибок ловится кодом. «Модель вернула
пустую строку вместо title» — это не нужно оценивать LLM-judge-ом,
это `title.length === 0`. «Budget_usd = "three thousand" вместо
3000» — это `typeof budget !== "number"`. Простые проверки дёшевы
и надёжны. Нет смысла платить за LLM-judge, чтобы он сказал «title
пустой» — safeParse это скажет бесплатно.

## 5. Бинарные лейблы vs Likert-шкала

Likert-шкала — это «оцени от 1 до 5». Кажется, что она даёт больше
информации: 4.2 vs 3.8 — можно строить графики, считать средние,
отслеживать тренды. Красиво на дашборде.

Проблема: **люди не согласны друг с другом на Likert-шкалах**.
Ты ставишь «4», твой коллега ставит «3» на тот же ответ. Кто прав?
Оба! Потому что разница между 3 и 4 — субъективна. Это называется
low inter-rater agreement.

Binary (pass/fail) решает эту проблему. Не «насколько хорошо?», а
«прошёл или нет?». Но для этого нужен **конкретный критерий**.

Плохо: «Оцени качество ответа» → что такое «качество»? Для одного
это полнота, для другого — краткость.

Хорошо: «Содержит ли ответ сумму в USD?» → да или нет. «Упомянут
ли дедлайн?» → да или нет. «Ответ на русском?» → да или нет.

Binary-лейблы форсируют тебя **быть конкретным** в том, что ты
проверяешь. Ты не можешь сказать «ну такое, на троечку» — ты
должен сформулировать бинарный критерий. И этот критерий потом
легко превращается в code assertion (уровень 1) или в чёткую
инструкцию для LLM-judge (уровень 2).

Ещё одна проблема Likert: LLM-judge, которому ты говоришь «оцени
от 1 до 5», склонен ставить 3-4 почти всегда. Распределение
оценок кучкуется в середине, и ты не видишь реальных проблем.
Binary — pass или fail, без «ну средне».

## 6. Error analysis

Error analysis — это самая недооценённая часть eval-процесса.

Что это: ты берёшь 30-50 реальных входов, прогоняешь через систему,
кладёшь input + output в таблицу (Google Sheets, Notion, или
самописный HTML-вьюер), и **читаешь руками**. Каждый ответ. Глазами.
В колонке рядом пишешь «что не так».

Зачем: потому что ты не знаешь, **какие** ошибки делает система,
пока не посмотришь. Ты можешь думать, что модель галлюцинирует
числа, а на самом деле она путает валюты. Ты можешь думать, что
она не понимает сарказм, а на самом деле она всегда добавляет «I'm
happy to help» в конце, и это раздражает пользователей.

Из чтения рождается **таксономия failure modes** — список категорий
ошибок:
- «В 8 из 50 случаев budget_usd не null, когда должен быть null»
- «В 5 из 50 случаев title длиннее 6 слов»
- «В 3 из 50 случаев deadline в ISO формате, хотя в тексте формат
  другой»

Каждая категория — кандидат на eval. Ты пишешь code assertion
для каждой, и теперь у тебя eval, основанный на **реальных**
ошибках, а не на воображаемых.

Почему ДО, а не ПОСЛЕ: если ты написал eval до error analysis,
ты тестируешь свои фантазии. «Наверное модель будет возвращать
невалидный JSON» — пишешь safeParse. Запускаешь — 10/10. Думаешь:
«всё хорошо». А модель на самом деле возвращает валидный JSON с
неправильными данными. Без error analysis ты не видишь реальных
проблем — ты видишь только то, что додумал сам.

В нашем случае: мы на Day 03 написали eval без error analysis
(потому что не было реальных пользователей), и на Day 04 первый
прогон content assertions показал, что модель была **права**, а наши
expected values — нет. Это мини-error analysis: мы прочитали 10
выходов, увидели паттерны (формат дат, перефразированные titles),
и поправили eval. В масштабе продукта — то же самое, но на 50-100
примерах.

Хамель советует: потрать день на собственный data viewer (простая
HTML-страница с input/output/лейбл), чтобы читать данные было
приятно. Если приятно — читаешь больше. Если неприятно (сырой JSON
в терминале) — читаешь три примера и бросаешь.

## 7. Первые итерации чинят eval, а не prompt

Это контринтуитивно. Ты строишь eval, чтобы найти баги модели.
Запускаешь — и видишь фейлы. Первая реакция: «модель ошибается,
надо переписать prompt». Но если ты посмотришь на фейлы
внимательно, в 70% случаев на первых итерациях проблема в eval:

**Wrong expected values.** Ты ожидал `budget_usd: 2000` для
диапазона $1500-$2500 (взял середину). Модель вернула `null`
(диапазон — не фикс). Кто прав? Модель — потому что в system
prompt написано «fixed budget». Ты неточно описал expected.

**Too-strict matching.** Ты ожидал `"2026-06-15"`, модель вернула
`"June 15, 2026"`. Exact match → fail. Но модель цитирует текст
как есть (prompt говорит «as stated»). Нужен contains-match, не
exact.

**Ambiguous cases.** Ты ожидал `"API migration project"`, модель
вернула `"Senior Backend API Migration"`. Кто прав? Оба. Title —
творческое поле, точного ответа нет. Нужен fuzzy match.

Это нормально. Eval — это code, и как любой code, он содержит
баги. Первые 2-3 итерации — дебаг eval-а. Только после того, как
eval стабилен (ты уверен, что expected values правильные и matching
адекватный), ты начинаешь видеть **реальные** баги модели — и тогда
уже итерируешь prompt.

В нашем случае за 3 запуска мы ни разу не трогали system prompt.
Все фиксы были в cases.ts (expected values) и eval.ts (matching
logic). И score вырос с 23/30 до 30/30.

## 8. Три способа получить structured JSON от Claude

### Способ 1: XML теги

Как работает: в system prompt пишешь «верни JSON в <json></json>
тегах». Claude возвращает текст с тегами. Ты вытаскиваешь JSON
regex-ом.

```typescript
// System prompt:
// "Return ONLY a JSON object wrapped in <json></json> tags."

const text = msg.content[0].text;
// text = '<json>{"title": "...", "budget_usd": 3000}</json>'

const match = text.match(/<json>([\s\S]*?)<\/json>/);
const json = JSON.parse(match[1].trim());
```

Плюсы: минимум кода, работает с любой моделью (Claude, GPT, Llama),
не зависит от SDK. Минусы: Claude может добавить текст вне тегов
(маловероятно, но парсинг опирается на regex, а не на протокол).

### Способ 2: Prefill (мёртв на Claude 4)

Как работало: ты добавлял в messages сообщение от assistant с
началом ответа `{`. Claude продолжал с этого места.

```typescript
messages: [
  { role: "user", content: "..." },
  { role: "assistant", content: "{" },  // prefill
]
// Claude продолжает: "title": "...", "budget_usd": 3000}
// Ты склеиваешь: "{" + response = полный JSON
```

Идея: если ответ уже начинается с `{`, модель «понимает», что
от неё ждут JSON, и не добавляет текст перед ним.

Почему мёртв: Claude Sonnet 4 возвращает HTTP 400: «This model
does not support assistant message prefill.» Anthropic убрали
эту возможность в четвёртом поколении моделей. Причина не
публична — возможно, это мешало новой архитектуре или создавало
проблемы с безопасностью.

Стоит ли учить: да, как историю. На собеседовании могут спросить.
Но использовать — нет.

### Способ 3: tool_use

Как работает: ты описываешь «инструмент» (tool) с JSON Schema для
аргументов. Говоришь Claude: «у тебя есть инструмент save_brief,
вот его схема, вызови его с данными из текста». Claude «вызывает»
инструмент — SDK возвращает аргументы как parsed объект.

```typescript
const msg = await client.messages.create({
  model: "claude-sonnet-4-6",
  tools: [{
    name: "save_brief",
    description: "Save extracted data.",
    input_schema: {
      type: "object",
      properties: {
        title: { type: "string" },
        budget_usd: { type: ["number", "null"] },
        deadline: { type: ["string", "null"] },
      },
      required: ["title", "budget_usd", "deadline"],
    },
  }],
  tool_choice: { type: "tool", name: "save_brief" },
  messages: [{ role: "user", content: "..." }],
});

const toolBlock = msg.content.find(b => b.type === "tool_use");
const data = toolBlock.input; // уже объект, не строка!
```

Claude **не вызывает функцию реально**. Он генерирует JSON-объект
с аргументами, API возвращает его в блоке `type: "tool_use"`, SDK
парсит автоматически. «Вызов» — метафора. Claude выбирает tool +
заполняет поля, исполнение — на твоей стороне.

`tool_choice: { type: "tool", name: "save_brief" }` — форсирует
Claude вызвать именно этот инструмент. Без этого он может решить
ответить текстом.

Плюсы: парсинг на уровне протокола (не regex), JSON Schema как
контракт, гарантированный tool_use блок в ответе.

Минусы: больше бойлерплейта (описание tool), описываешь схему
дважды (JSON Schema для API + Zod для валидации — решается
через `zod-to-json-schema`), чуть медленнее (Claude «думает»
про tool selection даже если tool один).

### Что выбирать

Для скриптов и экспериментов — XML теги. Просто, минимум кода.
Для продакшна — tool_use. Гарантия парсинга на уровне протокола,
не в твоём regex.

## 9. Что такое tool_use подробнее

Это одна из ключевых фич Claude API, и она используется для двух
совершенно разных задач:

### Задача А: Structured output (как мы делали)

Ты описываешь фейковый tool, у которого нет реальной реализации.
Цель — заставить Claude вернуть данные в нужной форме. Ты
описываешь schema → Claude заполняет поля → ты забираешь input.
Tool «save_brief» не сохраняет никуда — это просто способ
получить typed JSON.

### Задача Б: Реальные инструменты (будешь делать позже)

Claude может «решить» вызвать реальный инструмент: поиск в
базе данных, запрос к API, калькулятор. Ты описываешь tool с
реальной функцией. Claude вызывает его, ты исполняешь функцию,
возвращаешь результат Claude, он продолжает ответ.

Пример: Claude нужно узнать погоду. Ты описываешь tool
`get_weather(city: string)`. Claude отвечает: «мне нужно
вызвать get_weather("Buenos Aires")». Ты вызываешь реальный
API → получаешь «22°C» → отправляешь результат обратно Claude →
Claude говорит пользователю: «Сейчас в Buenos Aires 22°C».

Это цикл: user → Claude → tool_use → ты исполняешь → tool_result →
Claude → ответ пользователю. В коде это выглядит как ping-pong
между твоим сервером и API.

Для Day 05 мы использовали задачу А (structured output). Задача Б
(реальные инструменты) — это тема месяца 2 (research-agent).

### Что приходит в ответе API

Обычный ответ Claude — массив content blocks. Текстовый ответ
выглядит так:

```json
{
  "content": [
    { "type": "text", "text": "Here's the answer..." }
  ]
}
```

Когда Claude «вызывает tool», в ответе вместо (или рядом с)
text-блока появляется tool_use блок:

```json
{
  "content": [
    {
      "type": "tool_use",
      "id": "toolu_abc123",
      "name": "save_brief",
      "input": {
        "title": "SaaS Landing Page",
        "budget_usd": 3000,
        "deadline": "June 15, 2026"
      }
    }
  ]
}
```

`input` — уже объект, не строка. SDK сделал JSON.parse за тебя.
Это ключевое отличие от XML tags и prefill: парсинг на уровне
протокола.

## 10. Zod — зачем, если Claude возвращает JSON

JSON — это **формат**. Он говорит: «это объект с ключами и
значениями, строки в кавычках, числа без». JSON.parse() проверяет,
что строка — валидный JSON. Но не проверяет **содержимое**.

```typescript
JSON.parse('{"title": 123, "budget_usd": "три тысячи"}')
// Не упадёт! Это валидный JSON.
// Но title — число, а budget_usd — строка. Оба — баги.
```

Zod — это **контракт на содержимое**. Ты описываешь не просто
«тут объект», а «тут объект, в котором title — строка минимум
1 символ, budget_usd — число или null, deadline — строка или
null». И Zod проверяет каждое поле.

```typescript
const BriefSchema = z.object({
  title: z.string().min(1),       // строка, минимум 1 символ
  budget_usd: z.number().nullable(), // число или null
  deadline: z.string().nullable(),   // строка или null
});
```

`BriefSchema.safeParse(obj)` возвращает:
- `{ success: true, data: { title: "...", ... } }` — если всё ок
- `{ success: false, error: ZodError }` — если что-то не так,
  с подробным описанием каждой ошибки

Почему `safeParse`, а не `parse`: `.parse()` бросает exception
при ошибке. В eval нам нужно **записать ошибку** и продолжить
к следующему кейсу, а не упасть. `.safeParse()` возвращает
результат, который можно проверить if-ом.

Бонус: `z.infer<typeof BriefSchema>` — Zod генерирует TypeScript
тип из схемы. Не нужно писать `interface Brief { ... }` отдельно.
Один источник правды: схема Zod описывает и runtime-валидацию, и
compile-time типы.

Аналогия: JSON.parse() — это как проверить, что в конверте лежит
бумага. Zod — это как проверить, что на бумаге написано резюме
с именем, датой рождения и опытом работы, а не стихотворение.

## 11. Эмбеддинги

Эмбеддинг — это способ превратить текст в массив чисел, который
описывает «смысл» текста.

### Зачем

Компьютер не понимает текст. Он понимает числа. Если ты хочешь
найти «похожие» тексты, тебе нужен способ сравнить их
математически. Сравнивать строки посимвольно бесполезно: «как
вернуть товар?» и «какова ваша политика возврата?» — разные
строки, но один смысл.

Эмбеддинг-модель (например, sentence-transformers, OpenAI
text-embedding-3-small) берёт текст и выдаёт вектор — массив
из, скажем, 384 чисел с плавающей запятой:

```
"как вернуть товар?" → [0.12, -0.45, 0.78, ..., 0.33]  (384 числа)
"какова политика возврата?" → [0.11, -0.44, 0.77, ..., 0.34]  (384 числа)
"какая погода в Буэнос-Айресе?" → [0.91, 0.22, -0.15, ..., 0.67]  (384 числа)
```

Первые два вектора будут «близко» (похожий смысл). Третий —
«далеко» (другая тема).

### Как это используется

**RAG (Retrieval-Augmented Generation)** — основа твоего проекта 1
(chat-with-docs). Схема:

1. Берёшь документы (FAQ, статьи, docs), разбиваешь на куски.
2. Каждый кусок эмбеддишь → массив векторов.
3. Пользователь задаёт вопрос → эмбеддишь вопрос.
4. Находишь ближайшие куски (cosine similarity).
5. Отправляешь их в Claude вместе с вопросом: «Вот контекст,
   ответь на вопрос».

Без эмбеддингов ты не можешь сделать шаг 4 — «найти похожий
текст». С эмбеддингами — это поиск ближайшего вектора, быстрая
математическая операция.

### Что внутри вектора

Каждое число — это «вес» какого-то абстрактного смыслового
измерения. Измерение 47 может (условно) кодировать «про
финансы». Измерение 128 — «вопрос или утверждение». Измерение
300 — «формальный или разговорный». Мы не знаем, что именно
каждое измерение значит — это выучено моделью при обучении.
Но мы знаем, что похожие тексты дают похожие числа.

## 12. Cosine similarity

### Формула

```
cos(θ) = (A · B) / (|A| × |B|)
```

Три компонента:

**A · B (dot product)** — берёшь два вектора одинаковой длины,
умножаешь поэлементно, складываешь:

```
A = [1, 2, 3]
B = [4, 5, 6]
A · B = 1×4 + 2×5 + 3×6 = 4 + 10 + 18 = 32
```

Интуиция: dot product показывает, насколько векторы «согласны»
по каждому измерению. Если оба больше нуля по какому-то
измерению — вклад положительный. Если один положительный, другой
отрицательный — вклад отрицательный (они «спорят»).

**|A| (norm / длина вектора)** — корень из суммы квадратов:

```
|A| = √(1² + 2² + 3²) = √(1 + 4 + 9) = √14 ≈ 3.74
```

Зачем: нормализация. Длинный текст может давать вектор с большими
числами, короткий — с маленькими. Если сравнивать dot product
напрямую, длинные тексты будут «ближе» к длинным, а короткие к
коротким — просто из-за масштаба. Деление на произведение длин
убирает масштаб и оставляет только **направление**.

**cos(θ)** — косинус угла между векторами:
- **1** = угол 0° = векторы «смотрят» в одном направлении =
  тексты про одно и то же
- **0** = угол 90° = перпендикулярны = тексты не связаны
- **-1** = угол 180° = противоположные = тексты про
  противоположное (на практике для эмбеддингов текста значения
  редко бывают сильно отрицательными)

### Геометрическая интуиция

Представь 2D (два измерения вместо 384). Два вектора — это две
стрелки из начала координат. Cosine similarity — это косинус угла
между ними.

```
          ^ ↑ B (вектор "политика возврата")
         /
        / 15°  ← маленький угол → cosine ≈ 0.97 → ПОХОЖИ
       /
      ──────→ A (вектор "как вернуть товар")

          ^ C (вектор "погода в BA")
          |
          | 85°  ← большой угол → cosine ≈ 0.09 → НЕ ПОХОЖИ
          |
      ──────→ A
```

Маленький угол = высокий cosine = похожий смысл.
Большой угол = низкий cosine = разный смысл.

### В коде

```typescript
function dotProduct(a: number[], b: number[]): number {
  let sum = 0;
  for (let i = 0; i < a.length; i++) sum += a[i] * b[i];
  return sum;
}

function norm(v: number[]): number {
  return Math.sqrt(dotProduct(v, v)); // √(v·v)
}

function cosineSimilarity(a: number[], b: number[]): number {
  return dotProduct(a, b) / (norm(a) * norm(b));
}
```

## 13. Попарный cosine в FAQ eval

Задача: проверить, что модель отвечает **консистентно** — одно и
то же (по смыслу) на разные формулировки одного вопроса.

У тебя 3 варианта вопроса:
- «What's your return policy?»
- «How can I return an item?»
- «Wut's yur retrn polcy?» (с опечатками)

Все три — про одно. Если модель работает хорошо, ответы на все
три должны быть семантически похожи (пусть и по-разному
сформулированы).

Как проверяем:

1. Отправляем каждый вариант в Claude → получаем 3 ответа.
2. Эмбеддим все 3 ответа → 3 вектора.
3. Считаем cosine similarity между каждой парой:
   - ответ 0 ↔ ответ 1 → 0.94
   - ответ 0 ↔ ответ 2 → 0.91
   - ответ 1 ↔ ответ 2 → 0.93
4. Средний: (0.94 + 0.91 + 0.93) / 3 = 0.927

**0.927 — высокий.** Модель отвечает консистентно: неважно, как
сформулирован вопрос (чисто, с опечатками, длинно, коротко),
ответ семантически один и тот же.

Если бы средний cosine был 0.5 — это плохо. Значит модель
даёт **разные ответы** на один и тот же вопрос, в зависимости
от формулировки. Для FAQ-бота это катастрофа: пользователь
спрашивает «как вернуть?» — получает одно, спрашивает «return
policy?» — получает другое. Доверие к системе рушится.

Важный нюанс: cosine similarity сравнивает **смысл**, не текст.
Ответы могут быть дословно разными, но семантически одинаковыми —
cosine будет высоким. Это ровно то, что нам нужно: мы не хотим
идентичных строк, мы хотим одинакового смысла.

## 14. Time to first token (TTFT)

### Без стриминга

```
0s ──────── ждём ──────── 5.5s → ВЕСЬ ОТВЕТ СРАЗУ
          (пустой экран)         (текст появляется мгновенно)
```

Пользователь 5.5 секунд видит ничего. Потом бам — стена текста.
С точки зрения UX: 5.5 секунд — это вечность. Через 3 секунды
пользователь думает «сломалось». Через 5 — закрывает вкладку.

### Со стримингом

```
0s ── 0.7s → первое слово
             0.8s → второе слово
             0.9s → третье слово
             ...
             5.9s → последнее слово
```

Общее время даже чуть дольше (5.9s vs 5.5s — overhead на
стриминг-протокол). Но первое слово появляется через 0.7 секунды.
Пользователь видит, что система «думает» и уже «печатает».
Perceived latency — в 8 раз ниже.

### Почему TTFT важнее total time

Из UX-исследований (и из твоего 10-летнего фронтенд-опыта):
восприятие скорости определяется **первым визуальным откликом**,
а не общим временем. Прогресс-бар, который двигается, воспринимается
быстрее, чем спиннер, который крутится те же 5 секунд. Текст,
который «печатается» по словам, воспринимается быстрее, чем текст,
который появляется целиком после паузы.

В ChatGPT, Claude.ai, и любом другом LLM-чате — стриминг. Это
не опция, это minimum viable UX. Если ты строишь chat-with-docs
(проект 1) без стриминга, пользователь уйдёт к конкуренту.

### В коде

```typescript
const stream = client.messages.stream({ ... });

stream.on("text", (text) => {
  // вызывается каждый раз, когда Claude сгенерировал 1-3 токена
  process.stdout.write(text); // печатаем в терминал без переноса строки
});

const finalMessage = await stream.finalMessage();
// ждём конец стрима, получаем полный Message-объект с usage
```

`stream.on("text", ...)` — это event listener, как
`element.addEventListener("click", ...)`. Событие — «пришёл новый
кусок текста». Callback вызывается десятки раз за один ответ.
Frontend muscle memory переносится напрямую.
